# Confidential P2P Node 

**Zero-Trust Peer-to-Peer Architecture leveraging Intel SGX Hardware Isolation**

This repository contains the experimental implementation of an academic thesis project focused on securing distributed Peer-to-Peer (P2P) networks against **Insider Threats**. By shifting the Root of Trust from the host Operating System to the silicon hardware using **Intel Software Guard Extensions (SGX)**, this project demonstrates a highly resilient distributed architecture.

---

## Overview

In traditional P2P networks, security relies on a perimeter model: once a node authenticates, its traffic is generally trusted. However, if the host OS is compromised by an attacker (Insider Threat), this trust is entirely broken. 

This project implements a **Zero-Trust architecture** ("Never Trust, Always Verify") where:
- The Host OS is considered **Untrusted**.
- Cryptographic operations (key generation, AES-GCM encryption, MAC validation) are isolated within an **Intel SGX Enclave (Trusted)**.
- Compromised nodes attempting to bypass the Enclave and inject raw/tampered packets are cryptographically detected and dropped by honest peers.

---

## Architecture Design

The C application is strictly divided into two execution environments:

1. **Untrusted World (`node/main.c`)**: Runs in standard Linux user space. Handles networking, raw TCP/IP sockets, and the Command Line Interface (CLI).
2. **Trusted World (`enclave.signed.so`)**: Runs inside the SGX Processor Reserved Memory (PRM). Handles the validation and encryption of all P2P messages using AES-GCM.

Communication between the two worlds happens exclusively via predefined `ECALLs` (Enclave Calls) and `OCALLs` (Outside Calls) generated by `edger8r` and managed by the SGX driver.

---

## Threat Model & Simulation

This implementation includes a programmatic way to simulate a complete OS takeover (Insider Threat). Using Docker environment variables, a specific node can be instructed to act maliciously at runtime without altering the signed binary.

When activated via `NODE_BEHAVIOR=MALICIOUS`, the compromised node will bypass the local SGX Enclave entirely and attempt to inject unencrypted, raw payload strings directly into the TCP sockets targeting honest nodes. 

The honest receiving enclaves will deterministically drop the packets due to the absence of a valid hardware-generated Message Authentication Code (MAC).

## Getting Started (Docker Environment)

### Prerequisites
- Docker & Docker Compose
- Intel SGX hardware support (or SGX Simulation Mode enabled)

### Build and Run the P2P Mesh
The network is orchestrated using Docker Compose, creating an isolated custom bridge network (`172.20.0.0/16`) with static IPs for deterministic P2P routing.

```bash
# Clone the repository
git clone [https://github.com/filippo-zullo98/confidential-p2p-node.git](https://github.com/filippo-zullo98/confidential-p2p-node.git)
cd confidential-p2p-node

# Build and start the fleet in detached mode
docker compose up -d --build
```
## 🗡️ Reproducing the Insider Attack

To empirically validate the Zero-Trust architecture, follow these steps to simulate the attack:

### 1. Monitor the Honest Node (Victim)

Open a terminal and tail the logs of `node_b` to monitor its incoming traffic:

```bash
docker logs -f node_b
```

### 2. Attach to the Malicious Node (Attacker)

Open a second terminal and attach to `node_c` (pre-configured as an insider threat via `docker-compose.yml`):

```bash
docker attach node_c
```

*(Press `Enter` to reveal the `p2p-node>` CLI prompt).*

### 3. Launch the Attack

From the `node_c` prompt, send a tampered message to `node_b` (IP: 172.20.0.11, Port: 8080):

```plaintext
p2p-node> send 172.20.0.11 8080 Test_Attack
```

### Expected Result:

- In the attacker terminal (`node_c`), you will observe the system bypassing SGX and injecting raw garbage data.
- In the victim terminal (`node_b`), you will see the enclave immediately rejecting the packet due to a cryptographic integrity failure (Missing/Invalid MAC), proving the resilience of the hardware-backed Zero-Trust model.
